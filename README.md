## План автоматизации тестирования сценария перехода к форме записи и заполнения формы на сайте Нетологии (профессия «Инженер по тестированию»)

# LA-01. Перечень автоматизируемых сценариев
    
***1. Переход к странице профессии через меню «Каталог курсов»***

Шаги воспроизведения:
1. Открыть главную страницу сайта Нетологии.

**Ожидаемый результат: Страница загрузилась, виден основной контент и меню**

2. Навести курсор на пункт меню «Каталог курсов» и кликнуть по нему.

**Ожидаемый результат: Открылся каталог курсов**

3. В каталоге найти «Прагроммирование» и кликнуть по нему.

**Ожидаемый результат: Открыт раздел «Программирование», отображаются профессии/курсы этого направления**

4. В разделе найти и перейти на страницу профессии «Инженер по тестированию».

**Ожидаемый результат: Открыта страница профессии «Инженер по тестированию»**

5. Убедиться, что открыта именно страница указанной профессии

**Ожидаемый результат: URL страницы соответствует адресу страницы профессии, заголовок страницы содержит «Инженер по тестированию».**

6. Пролистать в низ и найти кнопку «Записаться» и кликнуть по ней.

**Ожидаемый результат: Кнопка найдена и нажата**

7. Убедиться, что форма записи отобразилась на странице либо открылась в модальном окне.

**Ожидаемый результат: Форма записи видна пользователю (либо открыто модальное окно с формой).**  

_Итоговый ожидаемый результат: пользователь успешно попал на форму записи через навигацию «Главная → Каталог курсов → Программирование → Профессия → Записаться»._


***2. Переход к странице профессии через контент главной страницы***

Шаги воспроизведения:

1. Открыть главную страницу сайта Нетологии.

**Ожидаемый результат: Страница загрузилась**

2. Найти блок с рекомендуемыми профессиями/баннер/карточку «Инженер по тестированию» в контенте главной страницы.

**Ожидаемый результат: Элемент карточки/баннера найден по тексту/атрибутам**

3. Кликнуть по ссылке/баннеру для перехода на страницу профессии.

**Ожидаемый результат: Переход выполнен, открыта страница профессии**

4. Проверить, что страница соответствует профессии .

**Ожидаемый результат: URL и заголовок страницы подтверждают, что открыта страница профессии «Инженер по тестированию»**

5. Кликнуть либо нажать кнопку «Тарифы».

**Ожидаемый результат: Открыт блок/страница с тарифами**

6. Убедиться в отображении формы записи.

**Ожидаемый результат: Форма записи отображается на странице (либо открыто модальное окно с формой).**

_Итоговый ожидаемый результат: пользователь попал на форму записи альтернативным путём через контент главной страницы._


***3. Проверка отображения и структуры формы записи (без отправки)***

Шаги воспроизведения:
1. Убедиться, что форма отображается (после клика «Записаться»)

**Ожидаемый результат: Форма видна, элементы рендерятся корректно**

2. Проверить наличие обязательных полей: «Имя», «Телефон»

**Ожидаемый результат: Все поля найдены по локаторам**

3. Проверить наличие подсказок/текстов валидации для каждого поля

**Ожидаемый результат: Подсказки видны при фокусе/ошибке (или в HTML-атрибутах)**

4. Убедиться, что кнопка отправки изначально неактивна (если логика формы это предусматривает)

**Ожидаемый результат: Кнопка не реагирует на клик до заполнения полей**

_Итоговый ожидаемый результат: все обязательные элементы присутствуют, клиентская валидация подготовлена_
 

***4. Позитивное заполнение формы (без отправки)***

Шаги воспроизведения:
1. Ввести валидное имя в поле «Имя»

**Ожидаемый результат: Поле заполнено, нет сообщений об ошибке**

2. Ввести корректный телефон в поле «Телефон»

**Ожидаемый результат: Поле заполнено, маска применена, ошибок нет**

3. Проверить отсутствие сообщений об ошибках валидации

**Ожидаемый результат: Ошибок нет, все поля в состоянии «OK»**

4. Убедиться, что кнопка отправки стала активной (если логика формы предусматривает)

**Ожидаемый результат: Кнопка готова к взаимодействию**

_Итоговый ожидаемый результат: при вводе валидных данных форма не показывает ошибок, кнопка отправки активна_


***5. Негативная проверка: пустое поле «Имя».***

Шаги воспроизведения:
1. Оставить поле «Имя» пустым (или очистить его)

**Ожидаемый результат: Поле пустое**

2. Убедиться, что отправка не происходит (если кнопка кликабельна)

**Ожидаемый результат: Для отправки заполните все поля**

_Итоговый ожидаемый результат: система требует заполнить поле «Имя»; отображается сообщение об ошибке; отправка не выполняется_


***6. Негативная проверка: некорректный телефон.***

Шаги воспроизведения:
1. Ввести некорректный телефон в поле «Телефон»

**Ожидаемый результат: Введите корректный номер телефона**

2. Убедиться, что отправка не происходит

**Ожидаемый результат: Для отправки заполните все поля**


_Итоговый ожидаемый результат: осистема требует заполнить поле «Номер телефона»; отображается сообщение об ошибке; отправка не выполняется._


# LA-02. Перечень используемых инструментов с обоснованием выбора
   Java + JUnit 5 — язык и фреймворк для написания автотестов. Подходит для учебных проектов, хорошо знаком пользователю, есть опыт работы.
   Selenide — надстройка над Selenium WebDriver. Упрощает работу с элементами страницы, сокращает код тестов, автоматически обрабатывает ожидания, удобен для обучения и быстрой разработки.
   Gradle — система сборки. Позволяет управлять зависимостями, запускать тесты, интегрироваться с CI. Опыт работы с Gradle у пользователя уже есть.
   Lombok — для сокращения шаблонного кода (геттеры, конструкторы и т.п.), чтобы сосредоточиться на логике тестов.
   Allure — для формирования наглядных отчётов по результатам тестов (сценарии, шаги, скриншоты при падении). Помогает быстро анализировать результаты.
   Git + GitHub — контроль версий и хранение кода тестов. Позволяет вести историю изменений, работать в команде, подключать CI.
   GitHub Actions — CI-пайплайн для автоматического запуска тестов. Даёт возможность регулярно проверять работоспособность сценариев при изменениях сайта.
   Faker — библиотека для генерации тестовых данных (имена, телефоны, email). Позволяет избежать использования реальных данных и упрощает подготовку тестовых наборов.

# LA-03. Перечень необходимых разрешений, данных и доступов
   Разрешение на проведение автоматизированного тестирования — письменное согласование с владельцем сайта (Нетология). Без него автоматизация на реальном сайте недопустима.
   Доступ к сайту Нетологии — публичный доступ (не требуется авторизация).
   Тестовые данные — фиктивные данные (имена, номера телефонов, email), генерируемые через Faker либо подготовленные вручную. Никаких реальных персональных данных использовать нельзя.
   Доступ к репозиторию GitHub — права на создание/редактирование репозитория для хранения кода автотестов.
   Доступ к настройке GitHub Actions — права на добавление workflow-файлов и запуск пайплайнов (или делегирование DevOps-специалисту).
   Доступы к инструментам разработки — установленный JDK 11, IntelliJ IDEA, Gradle, браузер (Chrome/Firefox) на машине разработчика или в CI-среде.

# LA-04. Перечень и описание возможных рисков при автоматизации
   Изменение структуры сайта — при обновлении дизайна/URL/классов элементов тесты перестанут находить элементы.
   Решение: использовать устойчивые локаторы (data-атрибуты, id), регулярно актуализировать сценарии, добавить мониторинг падений.
   Динамическая подгрузка контента — элементы могут появляться с задержкой, из‑за чего тесты будут падать на «элемент не найден».
   Решение: использовать явные ожидания (в Selenide они встроены), избегать жёстких Thread.sleep.
   Блокировка автоматизированных запросов — сайт может детектировать ботов и блокировать доступ.
   Решение: не запускать тесты слишком часто, соблюдать этику и правила сайта, иметь резервный план (например, локальный макет формы).
   Ложные срабатывания из‑за скорости интернета/нагрузки сервера — страницы могут загружаться дольше обычного.
   Решение: увеличить таймауты, добавить повторные попытки (retry), логировать скорость загрузки.
   Изменения в валидации формы — логика валидации может меняться (новые поля, другие правила).
   Решение: держать описание формы в документации, регулярно сверять с актуальной версией сайта.

# LA-05. Перечень необходимых специалистов для автоматизации
   Автоматизатор тестирования (QA Automation Engineer) — пишет и поддерживает автотесты, анализирует падения, актуализирует сценарии. Для учебного проекта эту роль может выполнять студент/стажёр под наставничеством.
   Наставник/Senior QA — ревью кода тестов, помощь в сложных случаях, контроль качества сценариев.
   DevOps‑инженер (опционально) — настройка CI/CD (GitHub Actions), контейнеров (Docker), управление окружением. На начальном этапе можно обойтись без него.
   Владелец продукта/аналитик (опционально) — уточнение требований к форме, подтверждение актуальности сценариев.

# LA-06. Интервальная оценка с учётом рисков в часах
   Анализ сайта и уточнение сценариев — 4–6 часов (изучение навигации, поиск всех способов попасть на страницу профессии).
   Подготовка окружения (JDK, IDE, Gradle, зависимости) — 2–3 часа (особенно если настраивается впервые).
   Написание базовых сценариев (переход к форме, проверка структуры) — 6–8 часов (2–3 часа на сценарий с учётом отладки).
   Реализация проверок валидации и негативных сценариев — 4–6 часов.
   Настройка CI (GitHub Actions) и Allure‑отчётов — 3–5 часов.
   Рефакторинг, добавление устойчивых локаторов, обработка ожиданий — 3–4 часа.
   Резерв на риски (изменения сайта, ложные падения, доработки) — 4–6 часов.
